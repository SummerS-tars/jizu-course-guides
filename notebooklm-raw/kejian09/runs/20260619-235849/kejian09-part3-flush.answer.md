根据**课件 09**及相关实验文档，关于 `beq` 分支导致的控制冒险与 Flush 操作讲解如下：

### 1. beq 分支与 PC 重定向时的 Flush 操作
在五级流水线中，分支指令（如 `beq`）默认在 **执行 (EX)** 阶段计算跳转目标并判断条件 [1, 2]。
*   **控制冒险**：当 `beq` 到达 EX 阶段确定需要跳转时，后续的两条指令（已进入 ID 和 IF 阶段）其实是错误路径上的指令 [3]。
*   **Flush 实现**：为了修正错误，硬件必须“冲刷”流水线。这通常通过 `Flush` 信号将处于 IF/ID 和 ID/EX 段寄存器中的指令清零，使其转变为 `nop`（空操作），同时将 PC 重定向至分支目标地址 [2, 4]。
*   **优化（课件 09 重点）**：通过在 **译码 (ID)** 阶段增加专门的相等比较器，可以将跳转决策提前。这样预测失败的代价将从清除 3 条指令减少到仅需清除 1 条指令（即仅 Flush 掉 IF 段的指令） [5, 6]。

### 2. 数值例：含分支指令序列的流水线状态展示
假设指令序列如下，且 `beq` 条件成立（跳转至地址 64）：
*   `20: beq $t1, $t2, 40` (目标地址 64)
*   `24: and $t0, $s0, $s1` (错误路径指令)
*   `64: slt $t3, $s2, $s3` (目标指令)

**逐步执行状态表（基于 ID 阶段完成判定 [5, 6]）：**

| 时钟周期 | 取指 (IF) | 译码 (ID) | 执行 (EX) | 访存 (MEM) | 写回 (WB) | 流水线动作说明 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **1** | `beq` | - | - | - | - | 取出 `beq` 指令 [7] |
| **2** | `and` | `beq` | - | - | - | `beq` 在 ID 判定成功，计算目标地址 64 [5] |
| **3** | `slt` (64) | **Flush** | `beq` | - | - | **Flush**：`and` 变为 `nop`；PC 指向 64 [6] |
| **4** | ... | `slt` | **nop** | `beq` | - | 目标指令进入译码，原错误指令消失 [8] |
