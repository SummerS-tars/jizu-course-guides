### 1. beq 分支与 Flush 操作原理
在 MIPS 五级流水线中，若要缩短分支延迟，通常将“转移地址计算”和“分支条件判断”提前到 **译码阶段 (ID)** [1, 2]。

*   **重定向**：当 ID 阶段检测到 `Branch=1` 且 `Zero=1` 时，分支成立（Taken），跳转目标地址被送往 PC [1]。
*   **Flush（冲刷）操作**：此时下一条顺序指令（PC+4 处的指令）已经由于取指惯性进入了 **取指阶段 (IF)** [3]。为了防止执行这条错误的指令，硬件会产生 `IF.Flush` 信号，将 IF/ID 流水段寄存器中的内容**清零**，使其变成一条 `nop` 指令 [1, 4]。

### 2. 数值例：流水线状态逐步填表
**指令序列示例**（来源：课件09-17 [2]）：
*   `20: beq $t1, $t2, 40` （假设 $t1=$t2，分支成立，跳转至 64）
*   `24: and $t0, $s0, $s1` （错误路径指令，应被冲刷）
*   `64: slt $t3, $s2, $s3` （跳转目标指令）

| 时钟周期 | IF (取指) | ID (译码) | EX (执行) | MEM (访存) | WB (写回) | 状态说明 |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| **Cycle 1** | `beq` | - | - | - | - | 取出 beq 指令 |
| **Cycle 2** | `and` | `beq` | - | - | - | **ID级判定跳转**，PC 重定向至 64 [2] |
| **Cycle 3** | `slt` | **`nop`** | `beq` | - | - | **Flush发生**：`and` 被清零 [2] |
| **Cycle 4** | ... | `slt` | `nop` | `beq` | - | 目标指令 `slt` 开始译码 |

### 3. 与 RISC-V Lab3 分支实现对照
根据实验要求与 **Lab3 报告** 的实现分析，两者存在显著差异：

| 特性 | 课件 09 优化模型 | RISC-V Lab3 实现 |
| :--- | :--- | :--- |
| **判定位置** | **ID 阶段**：通过专用比较器提前判定 [2]。 | **EX 阶段**：在执行级统一完成比较与判定 [5]。 |
| **Flush 代价** | **1 个周期**：仅需冲刷 1 条已取出的指令 [2]。 | **2-3 个周期**：需清空前级所有错误路径指令 [6, 7]。 |
| **设计考量** | 性能最优，但 ID 级逻辑复杂（需处理转发）[4]。 | 实现直接，可复用 EX 级已转发的操作数 [6]。 |
| **预测策略** | 动态/静态预测 [4]。 | 静态预测（默认不跳转，失败则冲刷）[6]。 |

**核心结论**：Lab3 报告指出，采用“EX 决断 + Flush”方案虽然性能代价稍高，但行为容易解释且与五级流水线结构高度兼容，是功能实现阶段的首选策略 [6]。
