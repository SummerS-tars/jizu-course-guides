期末考试中，**推测执行（Speculative Execution）与重排序缓冲区（ROB）**是考查动态调度进阶能力的核心。

### 1. 重排序缓冲区 (ROB) 结构与作用

ROB 是一种先进先出（FIFO）的缓冲队列，用于暂存已流出但尚未提交的指令结果 [1, 2]。它取代了 Tomasulo 算法中的部分寄存器换名功能，将指令的“执行完成”与“架构状态更新（提交）”解耦 [3, 4]。

**ROB 条目字段表：**

| 字段 | 作用 |
| :--- | :--- |
| **指令类型** | 标识该指令是分支、Store 或寄存器操作类 [3, 5] |
| **目标地址** | 存放结果的写回目标（如寄存器号或存储器地址） [3, 5] |
| **数据值** | 保存指令执行结果，直到指令正式提交 [3, 6] |
| **就绪字段** | 标识该指令是否已完成执行且数据已就绪 [3, 6] |

### 2. 顺序提交如何保证精确异常

动态调度允许**乱序执行**，但通过 ROB 实现**顺序提交（In-order Commit）**来保证精确异常 [7, 8]。

*   **执行逻辑**：指令乱序完成后，将结果写入 ROB 项而非直接修改寄存器堆 [9, 10]。
*   **提交逻辑**：只有当 ROB 队首的指令完成执行且无异常时，才允许将其结果写回架构寄存器或内存 [2, 9]。
*   **精确异常处理**：若队首指令触发异常，处理器会立即清空 ROB 中该指令及其之后的所有指令（这些指令虽可能已乱序执行，但因未提交，故未改变架构状态），确保异常现场与程序顺序执行时一致 [2, 8]。

### 3. 分支误预测时的回滚

推测执行允许处理器根据分支预测结果先行取指并执行后续指令。

*   **回滚机制**：当一条预测错误的分支指令到达 ROB 队首时，说明推测失败 [9, 11]。
*   **清空操作**：处理器会清空整个 ROB 队列并刷新寄存器状态表 [12, 13]。
*   **重新取指**：根据正确的分支目标地址重新开始取指执行 [9, 14]。

### 4. 数值例：ROB 状态变化演示

给定序列：`1. L.D F6, 0(R1)`；`2. ADD.D F0, F6, F2`；`3. BEQ ...`（预测跳转）。

| 周期 | 操作 | ROB 状态 (条目 1) | ROB 状态 (条目 2) | 说明 |
| :--- | :--- | :--- | :--- | :--- |
| **1** | **分配** | `Issue` (F6, 未就绪) | - | 指令 1 进入队尾 [15] |
| **2** | **分配** | `Execute` | `Issue` (F0, 等待#1) | 指令 2 依赖指令 1 的 ROB 编号 [16] |
| **3** | **执行** | `Write Result` (就绪) | `Execute` | 指令 1 完成，数据暂存 ROB [17] |
| **4** | **提交** | **已回收** | `Write Result` | 指令 1 到达队首并更新 F6 [18] |
