为您梳理了“指令流水线”中最易混淆的5组概念对比表：

### 1. 三大冒险对比
| 概念 | 产生原因 | 典型场景 | 解决对策 |
| :--- | :--- | :--- | :--- |
| **结构冒险** | 硬件资源冲突 [1, 2] | 访存与取指共用同一存储器 [3, 4] | IM/DM 分离、寄存器堆读写分离 [3, 5] |
| **数据冒险** | 指令间存在数据依赖 [6, 7] | RAW（写后读）：后面指令需用到未写回的结果 [8, 9] | 转发（旁路）、Load-Use 阻塞 [10-12] |
| **控制冒险** | 程序执行流改变 [2, 13] | 分支指令（beq）、跳转（j）、异常 [13, 14] | 静态/动态分支预测、缩短分支延迟 [10, 15, 16] |
**来源：** 课件06 [1, 3], Week7-8 [17], Lab1-3 [18]

### 2. 转发 vs. 阻塞
| 特性 | 转发 (Forwarding) | 阻塞 (Stalling) |
| :--- | :--- | :--- |
| **核心机制** | 提前把结果从段间寄存器送往 ALU 输入端 [19, 20] | 硬件让流水线暂停，等待数据写回 [7, 21] |
| **性能影响** | 不增加 CPI，性能最高 [22] | 引入空泡（Bubble），会增加延迟 [21, 23] |
| **适用性** | 解决大部分 ALU 指令引起的 RAW 冒险 [24, 25] | 解决无法通过转发处理的 Load-Use 冒险 [24, 26] |
**来源：** 课件06 [11, 24], Week7-8 [27, 28], Lab1-3 [18, 29]

### 3. Flush vs. Stall
| 特性 | 刷新 (Flush) | 阻塞 (Stall/Bubble) |
| :--- | :--- | :--- |
| **触发时机** | 分支预测失败或发生异常 [15, 30] | 检测到 Load-Use 冒险时 [31] |
| **操作对象** | 丢弃已经取进流水线的后续错误指令 [20, 32] | “冻结”当前及之前流水级，在后级插入空指令 [7, 14] |
| **硬件表现** | 将 IF/ID 等段间寄存器清零，变为 NOP [15, 32] | PC 和 IF/ID 保持原值，ID/EX 控制信号清零 [14, 33] |
**来源：** 课件06 [15, 32, 33], Week7-8 [28], Lab1-3 [18]

### 4. 吞吐率 vs. 加速比
| 指标 | 吞吐率 (Throughput) | 加速比 (Speedup) |
| :--- | :--- | :--- |
| **定义** | 单位时间内完成的指令条数 [2, 34] | 非流水线与流水线执行时间的比值 [34, 35] |
| **理想上限** | 每个时钟周期完成 1 条指令（CPI=1） [36] | 理论上等于流水线的段数 $m$ [35] |
| **侧重点** | 衡量系统整体的工作处理能力 [37] | 衡量流水线技术对单任务执行的提升效率 [38, 39] |
**来源：** 课件01 [34], 课件06 [36], Week7-8 [40]

### 5. 段间寄存器 vs. 通用寄存器
| 特性 | 段间寄存器 (Pipeline Register) | 通用寄存器 (GPR) |
| :--- | :--- | :--- |
| **透明性** | 对程序员不可见（透明） [41, 42] | 程序员可见，可通过指令读写 [43, 44] |
| **功能** | 暂存指令执行的中间状态和控制信号 [2, 45, 46] | 存放程序运行的变量和数据 [47, 48] |
| **更新频率** | 每个时钟周期都会无条件更新 [49] | 仅在 Write Back 阶段且 RegWr 有效时写入 [50, 51] |
**来源：** 课件06 [45, 46, 52], Week7-8 [53, 54]
