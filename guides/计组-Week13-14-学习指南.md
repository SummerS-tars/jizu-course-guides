## 0. 术语表

| 术语 | 大白话 |
|------|--------|
| **Cache Coherence（一致性）** | 多核对**同一地址**的读写，最终能读到「最新那份」 |
| **Memory Consistency（连贯性）** | 多核对**不同地址**操作的**全局可见顺序**约定（如 SC） |
| **Snooping（监听法）** | 各核盯着共享总线广播，自行判断是否失效/更新 |
| **Directory（目录法）** | 用目录记录「谁有这份副本」，点对点发消息 |
| **Write Invalidate（写作废）** | 写之前广播「作废」，其他副本标 Invalid |
| **Write Update（写更新）** | 写之后广播新数据，强制他人同步更新 |
| **MESI (Modified / Exclusive / Shared / Invalid)** | 四态缓存一致性协议 |
| **NUMA (Non-Uniform Memory Access)** | 内存分布在各节点，访存延迟「非均匀」 |

## 2. 核心知识

### 2.1 缓存一致性问题（Week 13）

> **本节要回答**：多核为何会出现不一致？共享总线架构的瓶颈是什么？

| 来源 | 位置 | 本节对应主题 |
|------|------|-------------|
| **课件 08** | 多核 Cache 不一致 | 写回导致旧值 |
| **唐朔飞** | **第 8 章** §8.1–8.2 | 一致性问题的提出 |
| **P&H RISC-V** | **第 6 章** §6.2 | 多核与 Cache |
| **课程记录** | `week13-周一-计组H.md` | 双核读写时序 |

**直觉**：每个核心有自己的小推车（私有 Cache），仓库（主存）只有一份货。厨师 A 改了配方只放在推车上、没送回仓库，厨师 B 仍从自己的推车拿旧货——数据视图就分裂了。（来源：w13-coherence-problem）

**经典时序**（写回策略）：

| 步骤 | 动作 | 结果 |
|------|------|------|
| 1 | 主存 X=5；P1、P2 各读入 Cache | 两核 Cache 均为 5 |
| 2 | P1 写 X=10 | 可能只更新 P1 Cache，主存仍为 5 |
| 3 | P2 再读 X | 若命中本地 Cache → 仍读到 **5**（不一致） |

```mermaid
graph TD
    subgraph P1["核心 P1"]
        C1["Cache: X=10"]
    end
    subgraph P2["核心 P2"]
        C2["Cache: X=5（旧）"]
    end
    Mem[("主存: X=5")]
    C1 -.->|修改未写回| Mem
    C2 -->|读命中旧值| C2
```

**共享总线瓶颈**（监听法前提）：核数增加 → 总线带宽争用、广播风暴、物理扇出受限；小规模 SMP（约 4–8 核）尚可，大规模需目录法。（来源：w13-coherence-problem）

> **小结 → 下一节**：问题明确后，硬件用**一致性协议**维护副本；首要分叉是**监听**还是**目录**。

### 2.2 监听法 vs 目录法（Week 13–14）

> **本节要回答**：两种协议如何工作？各自适用什么规模？

| 来源 | 位置 | 本节对应主题 |
|------|------|-------------|
| **课件 08** | 监听法、目录法 | 广播 vs 点对点 |
| **唐朔飞** | **第 8 章** §8.3 | 目录式与监听式 |
| **P&H RISC-V** | **第 6 章** §6.3 | Snooping |
| **课程记录** | `week13-周一`、`week14-周一-计组H.md` | 总线瓶颈、Mesh |

| 维度 | 监听法 (Snooping) | 目录法 (Directory) |
|------|-------------------|---------------------|
| 通信 | 共享总线**广播**，各 Cache 控制器监听 | **目录**记录块状态与持有者，**点对点**消息 |

| 缺点 | 总线带宽与监听频率成瓶颈 | 目录状态机复杂，需额外目录存储 |

（来源：w13-directory-vs-snoop、w14-mesi-numa）

### 2.3 写更新 vs 写作废（Week 13）

> **本节要回答**：两种写维护策略的原理、带宽开销与典型应用？

| 来源 | 位置 | 本节对应主题 |
|------|------|-------------|
| **课件 08** | 写更新、写作废 | 带宽对比 |
| **唐朔飞** | **第 8 章** §8.3.2 | 写策略 |
| **P&H RISC-V** | **第 6 章** §6.3 | Invalidate 主流 |
| **课程记录** | `week13-周一-计组H.md` | 写作废时序 |

| 策略 | 写时动作 | 带宽 | 典型场景 |
|------|----------|------|----------|
| **写更新** (Write Update) | 广播**新数据**，强制他人同步更新 | **高**（每次写都广播） | 读远多于写、对读延迟极敏感的场景 |
| **写作废** (Write Invalidate) | 广播**作废**，他人标 Invalid；读时再缺失获取 | **低**（连续写仅首次需总线事务） | **现代通用 CPU 主流**（MSI/MESI） |

**写作废时序**（来源：w13-write-update-invalidate）：

1. P1、P2 均持有 X=0（Shared）
2. P1 要写 X=1 → 先发总线 **Invalidate X**
3. P2 将本地 X 标为 **Invalid**；P1 获独占并写入 1
4. P2 再读 X → Cache miss → 从 P1 或主存取新值

> **直观理解**：写更新像「群发最新版文档」；写作废像「群发作废通知，谁要看再重新下载」——多数程序写少读多，后者更省总线。

> **小结 → 下一节**：写作废 + 监听/目录 → 具体状态机；课程重点为 **MESI**（MSI 加上独占态 E）。

### 2.4 MESI 协议与 NUMA（Week 14）

> **本节要回答**：四态各表示什么？何时转换？大规模多核如何扩展？

| 来源 | 位置 | 本节对应主题 |
|------|------|-------------|
| **课件 08**、**7a** | MESI、NUMA、互连 | 状态机、分布式 |
| **唐朔飞** | **第 8 章** §8.4 | MESI 协议 |
| **P&H RISC-V** | **第 6 章** §6.4–6.5 | 一致性协议 |
| **课程记录** | `week14-周一/周三-计组H.md` | 目录式、NUMA |

#### 2.4.1 先看问题：MESI 要给每个 Cache 行一张“状态身份证”

**MESI（Modified / Exclusive / Shared / Invalid，修改 / 独占 / 共享 / 无效）** 要解决的是同一 Cache line 在多个核心私有 Cache 中有多份副本时，硬件如何判断“这份数据能不能读、能不能直接写、主存是否还是最新”。它不是单纯背四个字母，而是给每个 Cache 行附一个状态，配合本地处理器事件（PrRd/PrWr）和总线监听事件（BusRd/BusRdX/Invalidate）迁移。（来源：w14-mesi-numa、课件 08）

**重点 / 难点 / 一般点区分**：

| 层级 | 内容 | 开卷优先级 |
|------|------|------------|
| **重点** | M/E/S/I 四态语义、本地读写、他人读写后的状态 | 必须能填状态表 |
| **难点** | M 态他人读时的 Flush/降级、S 态写前 BusUpgr、E vs M | 常见扣分点 |
| **一般点** | MOESI、NUMA 目录扩展 | 用于解释扩展性 |

**MESI 四态**（来源：w14-mesi-numa）：

| 状态 | 含义 |
|------|------|
| **M** Modified | 本 Cache **唯一**副本，且**已修改**（主存过期 Dirty） |
| **E** Exclusive | 本 Cache **唯一**副本，且与主存**一致**（Clean） |
| **S** Shared | 可能多核共有，均与主存一致 |
| **I** Invalid | 块无效 |

> **直观理解：** E 是“我独有且干净”，M 是“我独有但改脏了”，S 是“大家都有干净副本”，I 是“这张副本作废”。开卷题先判断“是否独占”和“主存是否最新”，再落到字母。

#### 2.4.2 状态迁移图：先分本地事件，再分监听事件

**转换直觉**：

- **PrRd 缺失**：无其他副本 → **E**；有其他副本 → **S**
- **PrWr**：在 **E/M** 可直接写（E→M）；在 **S** 需 BusUpgr 作废他人 → **M**
- **总线监听**：**M** 态见他人读 → 写回主存 → **S**；见他人写 → **I**

```mermaid
stateDiagram-v2
    [*] --> I
    I --> E : PrRd, 无其他副本
    I --> S : PrRd, 有其他副本
    E --> M : PrWr
    S --> M : PrWr + BusUpgr
    M --> S : 他人读, 写回
    M --> I : 他人写
    E --> I : 他人写
    S --> I : 他人写 / Invalidate
```

> **读图提示：** 图中 PrRd/PrWr 是本核心发出的处理器读/写；“他人读/他人写”是本 Cache 控制器在总线上监听到的事件。做题时先问“事件来自本地还是来自别的核”，再问“本行当前是否有其他副本”。不要把 E→M 的本地写和 S→M 的写作废混在一起：前者无总线事务，后者必须先让其他 S 副本失效。

**E vs M 易混**：E 是「独家且干净」，写不必发总线；M 是「独家且脏」，替换时需写回。（来源：w1314-mistakes-bridge）

#### 2.4.3 示例题：双核 MESI 状态迁移

**题目场景**：两个核心 P1/P2 共享地址 A，初值为 5，A 所在 Cache 行初始在两个 Cache 中都是 I。

**已知**：MESI 协议；写采用写作废；若 M 态被他人 BusRd 读到，拥有者需要 Flush（把最新数据提供到总线/主存）并降级为 S。

**求**：下面四步后 P1/P2 状态、总线事务和数据来源。

| 步骤 | 动作 | P1 状态 | P2 状态 | 总线事务 | 数据来源 |
|------|------|:------:|:------:|----------|----------|
| 1 | P1 读 A | E | I | BusRd | 主存 5 |
| 2 | P2 读 A | S | S | BusRd | 主存或 P1 |
| 3 | P1 写 A=10 | M | I | BusUpgr / Invalidate | P1 本地修改 |
| 4 | P2 读 A | S | S | BusRd + Flush | P1 提供 10，主存同步 |

**步骤解释**：

1. P1 首次读且无人持有，进入 E，因为它独占且干净。
2. P2 再读同一行，P1 不再独占，双方为 S。
3. P1 在 S 态写前必须让 P2 的副本失效，然后自己进入 M。
4. P2 读到 P1 的 M 态行时，P1 必须提供最新值并降级为 S。

**结果解释**：一致性保证 P2 第 4 步读到 10，而不是主存旧值 5；代价是总线事务和可能的写回/降级。

> **易错提醒：** 若第 1 步 P1 读后是 E，P1 随后本地写可 E→M，无需总线；只有已有共享副本 S 时，写才需要 BusUpgr/Invalidate。

#### 2.4.4 NUMA 与目录：为什么不能一直靠广播

**NUMA 与分布式架构**：

- 存储器物理分布在各节点（核 + 私有 Cache + 本地内存分片），片内 Mesh/Crossbar 互连
- **NUMA**：访问本地内存快，远程内存慢 → 数据应靠近使用它的核
- 非总线互连下监听失效 → **目录式一致性**用共享向量只通知相关核，消除广播风暴（来源：w14-mesi-numa）

> **小结 → 下一节**：一致性保证同地址最新值；Week 15 将 I/O/DMA 纳入「谁与主存/Cache 同步」的更大图景。

### 2.5 连贯性模型与 Fence（Week 14 补强）

> **本节要回答**：Coherence 已保证同一地址最新，为什么还需要 Consistency 和 Fence？

| 来源 | 位置 | 本节对应主题 |
|------|------|-------------|
| **课件 08** | 存储连贯性、Fence、同步原语 | SC、弱模型、屏障 |
| **唐朔飞** | **第 8 章** §8.4 | 多核访存顺序 |
| **P&H RISC-V** | **第 6 章** §6.3–6.5 | Memory consistency、同步 |
| **课程记录** | `week14-周三-计组H.md` | Fence 与一致性边界 |

#### 2.5.1 先分清两个“一致性”

**Coherence（缓存一致性）** 管同一个地址：读 A 最终应看到对 A 的最新写。**Consistency（存储连贯性）** 管不同地址之间的全局可见顺序：A、B 两个地址的写入，别的核按什么顺序观察到。前者由 MESI/目录等硬件协议维护，后者由内存模型和 Fence 约束。（来源：课件 08 deep raw，复用 `kejian08-partC-consistency`）

| 维度 | Coherence（一致性） | Consistency（连贯性） |
|------|----------------------|------------------------|
| 关注地址 | 同一地址 / 同一 Cache line | 多个地址之间的顺序 |
| 常见题型 | MESI 状态迁移 | SC 反例、Fence 放置 |
| 典型机制 | 写作废、监听/目录 | `fence`、`.aq`、`.rl`、SC/弱模型 |

#### 2.5.2 反例：每个地址都一致，整体顺序仍可能乱

**题目场景**：两个核心各自先写一个变量，再读对方变量；初值 A=B=0。

**已知**：

```text
P1: A = 1; R1 = B;
P2: B = 1; R2 = A;
```

**求**：弱内存模型下是否可能出现 R1=0 且 R2=0？为什么 SC 下不允许？

**步骤**：

1. 若采用顺序一致性 SC（Sequential Consistency，顺序一致性），每个核内部程序序必须保留，所有访存可合并成一个全局序。
2. 在任意 SC 全局序里，若 P1 先读到 B=0，说明 P2 的 `B=1` 还没全局可见；若 P2 又读到 A=0，说明 P1 的 `A=1` 也还没全局可见，这与两边都已经执行“先写后读”的程序序冲突。
3. 弱模型允许写缓冲或读写重排：本核后续读可先执行，而本核写还滞留在写缓冲中，于是两个读都可能看到旧值 0。

**结果解释**：R1=0 且 R2=0 不是 MESI 没维护同一地址，而是 Consistency 没强制跨地址顺序。

> **易错提醒：** “读到了旧 B”不能直接推出 B 的 Coherence 失效；如果 B 的写尚未按内存模型对 P1 可见，问题属于连贯性 / 内存序。

#### 2.5.3 Fence 放在哪里：普通数据同步 vs 页表同步

下面的图解决“生产者写数据和写 flag 之间为什么要有屏障”。图中 Fence 表示普通内存屏障或 RISC-V 原子指令上的 acquire/release 语义。

```mermaid
sequenceDiagram
    participant P as 生产者
    participant M as 共享内存
    participant C as 消费者
    P->>M: Data = NEW
    Note over P: fence 或 release，禁止 Data 晚于 Flag 可见
    P->>M: Flag = Ready
    C->>M: 读 Flag (acquire)
    C->>M: Read(Data)
```

> **读图提示：** 生产者侧屏障保证临界区内写入先于“发布 flag”可见；消费者侧 acquire 保证看到 flag 后，再读 Data 不会被重排到获锁/读 flag 之前。`SFENCE.VMA` 不在这张普通数据同步图里，因为它管的是页表/TLB 翻译缓存。

| 指令 / 语义 | 管什么 | 不能替代什么 |
|-------------|--------|--------------|
| `fence rw,rw` | 普通 load/store 与 I/O 访存顺序 | 不能刷新 TLB 旧映射 |
| `.aq` / `.rl` | 原子指令前后的 acquire/release 顺序 | 不能单独修正页表翻译 |
| `SFENCE.VMA` | PTE/`satp` 修改后的 TLB 翻译结果 | 不能修复普通数据竞争 |

> **小结 → 下一节**：MESI 是同地址副本协议；Fence/Consistency 是跨地址顺序契约；Lab 相关内容只保留 `SFENCE.VMA` 轻量引用，具体实验路径详见后续 Lab 专章。

## 7. 追问块

> **追问 1**：若系统采用**写更新**而非写作废，P1 连续 100 次写同一变量，总线上会发生什么？与 MESI 下写作废对比如何？
>
> **答**：写更新会广播 **100 次新数据**，总线带宽爆炸。MESI 写作废下首次写发 Invalidate，后续 99 次若他人已 Invalid 则**无需总线事务**——写密集场景带宽优势巨大。

> **追问 2**：某块处于 **E** 态，P1 执行 PrWr 后状态？若此时 P2 在总线上发起对该块的读，P1 需要哪些动作？
>
> **答**：PrWr 后 **E→M**（独占脏）。P2 总线读时，P1 监听须 **写回主存**（若 M）并降为 **S**，或从 M 直接供给数据后转 **S**——保证 P2 读到最新值。

> **追问 3**：一致性（Coherence）保证了「读 X 能见到最新 X」，为何仍可能出现「线程 A 写 X、写 Y，线程 B 先看到 Y 再看到 X」？这与哪个概念有关？
>
> **答**：这是 **Memory Consistency（连贯性）** 问题——一致性不管**不同地址**间的全局可见顺序。宽松模型下写缓冲/乱序可导致 B 观察到与程序序不同的交叉；需 Fence 或 SC 模型约束。

> **直观理解**：把 MESI 想成「房产证」——**S** 是多人共有产权；**E** 是你独家持有且与登记簿一致；**M** 是你独家持有但已改建未报备；**I** 是产权证作废，下次交易须重新办证。
