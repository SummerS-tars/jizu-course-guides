## 0. 术语表

### 2.1 虚拟存储（Week 10）

> **本节要回答**：为什么需要虚存？逻辑地址、按需分页、页表各是什么？

| 来源 | 位置 | 本节对应主题 |
|------|------|-------------|
| **课件 07** | 虚存、存储墙 | 逻辑地址空间、页表 |
| **唐朔飞** | **第 7 章** §7.3 | 虚存基本概念 |
| **P&H RISC-V** | **第 5 章** §5.6 | 虚存动机 |
| **课程记录** | `week10-周三-计组H.md` | 存储墙、按需分页 |

**直觉**：每个进程拿到一张「额度很大的信用卡」（独立逻辑地址空间），真正访问时才从物理内存「拨款」；未用到的页留在磁盘。（来源：w10-virtual-memory）

| 概念 | 要点 |
|------|------|
| 存储墙 | 带宽/延迟跟不上算力；层次化存储缓解 |
| 逻辑地址空间 | 每进程独立、连续；简化链接加载 |
| 按需分页 | 缺页时才从磁盘调入 |
| 页表 | VPN→PPN 映射字典；含 V/R/W/X/U/A/D 等位 |

```mermaid
graph TD
    K[内核空间] --> S[用户栈]
    S --> L[共享库]
    L --> H[堆]
    H --> D[数据段]
    D --> C[代码段]
```

> **小结 → 下一节**：虚存提供大地址空间与隔离；硬件用 **多级页表** 把 VA 译成 PA——Sv39 是 RISC-V 课程与 Lab5 的实现标准。

### 2.2 Sv39 三级页表（Week 10–11）

> **本节要回答**：39 位虚址怎么拆？PTE 各位什么意思？怎么手算一次地址转换？

| 来源 | 位置 | 本节对应主题 |
|------|------|-------------|
| **课件 07** | Sv39、PTE 位域 | VPN 划分、Page Walk |
| **唐朔飞** | **第 7 章** §7.3.2 | 页表项结构 |
| **P&H RISC-V** | **第 5 章** §5.7 | RISC-V 分页 |
| **课程记录** | `week10-周三-计组H.md` | 地址转换手算 |

#### 2.2.1 先看问题：为什么 Sv39 要拆成 9+9+9+12？

**Sv39（Supervisor Virtual Addressing 39-bit，RISC-V 39 位监管者虚拟地址方案）** 要解决的是“进程看到的虚拟地址 VA（Virtual Address，虚拟地址）如何低成本译成物理地址 PA（Physical Address，物理地址）”。如果只用一级页表，39 位有效地址会产生巨大的页表空间；三级页表把 VPN（Virtual Page Number，虚页号）分层，未用到的地址区间可以不分配下级页表。（来源：w10-sv39-page-table、课件 07）

**重点 / 难点 / 一般点区分**：

| 层级 | 内容 | 开卷优先级 |
|------|------|------------|
| **重点** | VA 拆位、`satp.PPN` 找根页表、叶子 PTE 生成 PA | 必须能手算 |
| **难点** | 叶子/非叶子 PTE、巨页、权限位触发 Page Fault | 常出辨析题 |
| **一般点** | WARL、ASID 位宽、实现细节 | 了解用途即可 |

**Sv39 虚址划分**（39 位有效，高位符号扩展；表头 VPN/PPN 已在术语表解释为虚页号/物理页号）：

| 位域 | 含义 |
|------|------|
| [38:30] | VPN[2] 根页表索引 |
| [29:21] | VPN[1] |
| [20:12] | VPN[0] |
| [11:0] | 页内偏移（4KB） |

> **边界说明：** Sv39 的 64 位 VA 不是全部参与翻译，真正拆的是低 39 位；高位必须是第 38 位的符号扩展，否则属于非法地址。4KB 页的页内偏移固定 12 位，因此每级 VPN 正好 9 位，对应一页中 512 个 8 字节 PTE。

#### 2.2.2 PTE 位域：先判断“叶子”，再判断权限

**PTE（Page Table Entry，页表项）关键标志位**：V 有效；R/W/X 权限（全 0 表示指向下一级）；U 用户可访问；A 已访问；D 脏页。（来源：w10-sv39-page-table）

判题时不要先背位名，而要按用途分三类：

| 类别 | 位 | 用途 |
|------|----|------|
| 有效性 | V | V=0 直接 Page Fault，不再继续 Walk |
| 叶子判定 | R/W/X | 至少一个为 1 表示叶子 PTE；全 0 表示非叶子，PPN 指向下一级页表 |
| 权限与状态 | U/A/D、SUM/MXR | 判断用户/监管者能否访问、读可执行页、脏页写入等边界 |

> **易错提醒：** “找到 PTE”不等于“转换成功”。还要检查 V、R/W/X、U、A/D 以及当前特权级；权限错和 V=0 都是 Page Fault，而不是 TLB miss。

#### 2.2.3 Page Walk 图：从 `satp.PPN` 到叶子 PTE

下面的图要解决“硬件拿到 VA 后，每一级页表地址从哪里来”。图中 `satp.PPN` 是根页表物理页号；每个 PTE 占 8 字节，所以每级 PTE 地址都形如 `页表基址 + VPN[i] × 8`。图节点中的 PTW 指 Page Table Walk（页表遍历）。

**Page Walk 步骤**：

1. 从 `satp.PPN` 得根页表物理基址
2. 用 VPN[2] 索引读 PTE2 → 若非叶子则 PPN 作 L1 基址
3. 同理 L1→L0
4. `PA = LeafPTE.PPN << 12 | offset`

**手算例**（来源：raw）：`satp.PPN=0x80000`，`VA=0x8_0000_2ABC` → 逐级索引得 `PA=0x9_0000_0ABC`。

```mermaid
flowchart TD
    VA[虚拟地址 VA] --> VPN2[VPN2 索引根页表]
    VPN2 --> PTE2[PTE2]
    PTE2 --> VPN1[VPN1 索引 L1]
    VPN1 --> PTE1[PTE1]
    PTE1 --> VPN0[VPN0 索引 L0]
    VPN0 --> LEAF[叶子 PTE 得到 PPN]
    LEAF --> PA[PA = PPN 左移 12 位<br/>再拼接 offset]
    SATP[satp.PPN] --> VPN2
```

> **读图提示：** 先从 `satp.PPN` 读根页表物理基址，再沿 VPN[2]、VPN[1]、VPN[0] 逐级索引。若中途读到非叶子 PTE，就把该 PTE 的 PPN 当作下一级页表基址；若读到叶子 PTE，就把叶子 PPN 与 VA 的 offset 拼成 PA。图里没有画权限检查，但实际硬件必须在每次读 PTE 后检查 V/R/W/X/U/A/D 等位。

#### 2.2.4 示例题：Sv39 地址转换手算模板

**题目场景**：操作系统已把当前进程根页表写入 `satp`，CPU 访问某条 load 指令给出的 VA，需要判断最终 PA。

**已知**：`satp.PPN=0x80000`，`VA=0x8_0000_2ABC`；三级页表遍历最终读到叶子 PTE，其 PPN 为 `0x90000`。（来源：raw，原例题本轮补齐题面）

**求**：该 VA 对应的 PA，并说明哪些检查会导致 Page Fault。

**公式**：

$$\text{PTEAddr}_i = (\text{CurrentPPN} << 12) + \text{VPN}[i] \times 8$$

$$\text{PA} = (\text{LeafPTE.PPN} << 12) \;|\; \text{offset}$$

**步骤**：

1. 先拆 VA：低 12 位 `0xABC` 是页内偏移；其余位按 VPN[2]/VPN[1]/VPN[0] 各 9 位拆。
2. 用 `satp.PPN << 12` 得到根页表基址，再用 VPN[2] 找 PTE2。
3. 若 PTE2 为非叶子，取 PTE2.PPN 作为 L1 基址；重复处理 VPN[1]、VPN[0]。
4. 读到叶子 PTE 后，检查 V=1、R/W/X 权限匹配、A/D 等状态合法。
5. 拼接 PA：`0x90000 << 12 | 0xABC = 0x9_0000_0ABC`。

**结果解释**：PA 的高位来自叶子 PTE 的 PPN，低 12 位 offset 保持不变；这体现“页内偏移不参与页表查找”。

> **易错提醒：** 2MB 巨页的叶子 PTE 会出现在 L1，此时 VPN[0] 并入页内偏移；1GB 巨页的叶子在 L2。不要机械地认为所有转换都必须走满三级。

> **小结 → 下一节**：每次 Page Walk 要多次访存；**TLB + SATP** 缓存最近映射，并靠 SFENCE.VMA 在映射变更时保持一致。

### 2.3 SATP、TLB 与 SFENCE.VMA（Week 11）

> **本节要回答**：SATP 三个字段？TLB 干什么？四种 SFENCE 范围？

| 来源 | 位置 | 本节对应主题 |
|------|------|-------------|
| **课件 07** | TLB、地址转换加速 | SATP、SFENCE |
| **唐朔飞** | **第 7 章** §7.3.3 | TLB 原理 |
| **P&H RISC-V** | **第 5 章** §5.7 | TLB 与页表遍历 |
| **课程记录** | `week11-周一/周三-计组H.md` | Lab4 SATP、ASID |

#### 2.3.1 `satp`：不是普通地址，而是地址空间入口

**SATP（Supervisor Address Translation and Protection，监管者地址转换与保护寄存器）** 负责告诉硬件“是否开启分页、当前地址空间是谁、根页表在哪”。它不是某个普通内存地址，而是地址翻译机制的入口 CSR（Control and Status Register，控制状态寄存器）。

| 字段 | 位 | 含义 |
|------|-----|------|
| MODE | 63–60 | 0=Bare，8=Sv39 |
| ASID | 59–44 | 区分进程，减少切换刷 TLB |
| PPN | 43–0 | 根页表物理页号 |

**WARL**：可写任意值，读回仅合法值（如不支持的 Mode 被忽略）。（来源：w11-satp-tlb-sfence）

> **边界说明：** 修改 `satp` 相当于切换地址空间入口。流水线中已经取到、译到的指令可能仍按旧映射继续走，因此需要冲刷流水线；TLB 中旧 VPN→PPN 映射也需要用 `SFENCE.VMA` 失效。

#### 2.3.2 TLB：缓存的是翻译结果，不是数据块

**TLB（Translation Lookaside Buffer，地址转换后备缓冲）** 缓存最近的 VPN→PPN 和权限信息；命中则跳过内存中的多级遍历。它解决的是“页表太慢”，不是“数据不在 Cache”。

| 事件 | 含义 | 处理主体 |
|------|------|----------|
| TLB hit | 找到可用翻译 | 硬件直接拼 PA |
| TLB miss | 快表没有该翻译，但页表可能有效 | 硬件 PTW 或软件 refill |
| Page Fault | PTE 无效、权限错、A/D 不满足等 | 进入 OS 异常处理 |

#### 2.3.3 `SFENCE.VMA`：页表改完后清翻译缓存

**SFENCE.VMA 四种范围**：

| rs1（虚拟地址 VA 选择） | rs2（ASID 选择） | 失效范围 |
|-----|-----|----------|
| x0 | x0 | 全部 |
| x0 | ≠x0 | 指定 ASID 全部 |
| ≠x0 | x0 | 指定 VA 全局 |
| ≠x0 | ≠x0 | 指定 ASID + VA |

ASID 耗尽回收后必须 `SFENCE.VMA` 同步。

下面的图要回答“为什么只改 PTE 内存还不够”。实线是正常翻译路径，虚线是 OS 改页表后的维护动作。

```mermaid
sequenceDiagram
    participant OS as OS/内核
    participant PT as 页表内存 PTE
    participant TLB as TLB 翻译缓存
    participant CPU as CPU 取指/访存
    OS->>PT: 修改 PTE 或 satp
    OS-->>TLB: SFENCE.VMA 失效旧翻译
    CPU->>TLB: 后续 VA 查询
    TLB-->>CPU: 重新 Page Walk 得新 PPN/权限
```

> **读图提示：** `SFENCE.VMA` 不负责把普通数据写回主存，也不替代多核数据同步的 `fence`；它只约束页表修改与后续地址翻译之间的顺序。若表中 rs1/rs2 都是 x0，范围最大；若能给出具体 VA 和 ASID，失效范围更小。

#### 2.3.4 示例题：TLB miss 与 Page Fault 判别

**题目场景**：某 load 指令访问 VA，TLB 未命中，硬件开始 Sv39 Page Walk。

**已知**：L2、L1 PTE 均有效且为非叶子；L0 PTE 的 V=1、R=1、W=0、U=1、A=1、D=0。当前处于 S 模式，`mstatus.SUM=0`，该指令试图读用户页。

**求**：这是 TLB miss 还是 Page Fault？

**步骤**：

1. TLB 未命中只说明快表没有翻译，不能直接判 Fault。
2. Page Walk 找到 L0 叶子 PTE，V=1 且 R=1，读权限本身允许。
3. 但 PTE.U=1 表示用户页；当前是 S 模式且 SUM=0，S 模式不允许访问用户页。
4. 因此最终触发 Page Fault，而不是单纯 TLB miss。

**结果解释**：TLB miss 是“缓存未命中”；Page Fault 是“页表或权限不允许”。二者可能先后发生：先 miss，Walk 后才发现 Fault。

> **小结 → 下一节**：虚存解决容量与隔离；Week 12 用 **Cache** 解决译址后的访存速度，并与多核存储模型衔接。

## 4. 易混淆概念

| 对比组 | 正确理解 |
|--------|----------|
| VA vs PA | CPU 用 VA，MMU 译成 PA 才访存 |
| 页 vs 页框 | 虚存划分单位 vs 物理内存划分单位 |
| TLB miss vs Page Fault | 前者是快表没有缓存翻译，仍可 Page Walk；后者是页表无效、权限错或 A/D 等状态不满足，需要异常处理 |
| SFENCE 全局 vs 局部 | 见 §2.3 四组合 |

## 7. 追问块

> **追问 1**：若 `satp` 在运行中被修改，流水线中已取指、已译码的指令会怎样？为什么必须 flush？
>
> **答**：已取指/译码指令仍按**旧页表**翻译地址，会继续访问错误物理页。修改 `satp` 后必须 **冲刷流水线** 并从新 PC 重新取指，保证后续指令在新地址空间下执行——Lab5 trap 与 CSR 写后 flush 同源。

> **追问 2**：2MB 巨页与 4KB 页在 VPN 划分上有何不同？
>
> **答**：巨页叶子 PTE 出现在 **L1**（VPN[0] 并入页内偏移），只需两级 Walk；4KB 页叶子在 **L0**，需三级 Walk。巨页 TLB Reach 更大但可能浪费物理页内部空间。

> **追问 3**：TLB miss 但页在内存中，与 Page Fault 的处理有何不同？
>
> **答**：TLB miss 仍可由硬件 **Page Walk** 完成（多次内存访问）；Page Fault 是 PTE 无效或权限违例，须 **陷入 OS** 调入页或杀进程——延迟量级差几个数量级。
